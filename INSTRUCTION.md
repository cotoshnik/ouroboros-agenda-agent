# ИНСТРУКЦИЯ: запуск и работа с агентом подготовки аналитических справок

Версия 2 (v2). Проект Sber AI Hack — автономный агент на базе Ouroboros.
Документ описывает всё: от установки до ежедневной работы, включая
данные MCP-сервера, все инструменты, форматы файлов и типичные проблемы.

---

## 1. Что это такое

Агент `agenda_briefing_agent` автоматизирует подготовку аналитических
справок дежурной смены:

1. **Сам обходит открытые веб-источники** (RSS-ленты: РБК, РИА Новости,
   ТАСС, Интерфакс, Банк России), собирает публикации, размечает их по
   темам повестки и выделяет экстренные сигналы.
2. **Принимает внутренние показатели банка** (67 показателей из каталога:
   клиентопоток, наличность, КТД, инциденты и т.д.) — они поступают из
   внутренних источников (письма, инфопанель) и загружаются через
   инструмент `save_observation`.
3. **Собирает черновик аналитической справки** одним вызовом: базовый
   блок (49 обязательных показателей) + динамический блок (по наличию
   данных) + сигналы + раздел веб-мониторинга + отчёт о полноте данных.
4. Финальную верификацию, выводы и отправку делает **человек-аналитик**
   (human-in-the-loop). Агент ничего никуда не отправляет сам.

## 6. Подключение сервера в Ouroboros

Путь в интерфейсе: **Settings → Advanced → MCP** → добавить сервер.

Заполните поля формы:

| Поле | Значение |
|------|----------|
| **Server ID / Name** | `agenda_briefing` |
| **Display name** | `Аналитические справки (локально)` — любая подпись |
| **Server URL** | `http://localhost:9999/mcp` |
| **Transport** | `streamable_http` |
| **Allowed tools** | оставить пустым (= все) или перечислить 9 штук через запятую: `list_indicators, get_selection_criteria, save_observation, get_observations, fetch_web_sources, get_signals, check_completeness, detect_hot_topics, build_report_draft` |

Либо вставьте готовый JSON из файла `mcp/mcp_servers.json`:

```json
{
  "name": "agenda_briefing",
  "url": "http://localhost:9999/mcp",
  "transport": "streamable_http"
}
```

После сохранения Ouroboros **сам обнаружит инструменты** сервера — в чате
агент сможет их вызывать. Проверка: спросите агента «какие MCP-инструменты
тебе доступны?» — должен перечислить 9 штук.

## 7. Данные MCP-сервера — полное описание

### 7.1. Каталог показателей — `memory/knowledge/indicators.json`

Единый источник истины по показателям. 67 записей:

```json
{
  "id": 26,
  "name": "Коэффициент технической доступности, %",
  "base_block": true,
  "dynamic_block": false,
  "selection_criterion": "Значение КТД берется за прошедшие сутки (письмо на почту присылается ориентировочно в 07:00-07:30)"
}
```

- `base_block: true` — показатель **обязателен** в каждой справке (49 шт.);
- `dynamic_block: true` — включается **при наличии данных** (18 шт.);
- `selection_criterion` — окно отсечки и источник (за какие сутки брать,
  во сколько приходит письмо). Агент обязан соблюдать эти окна.

### 7.2. Внутренние показатели — `data/observations_<дата>.json`

Значения за конкретные сутки. Создаётся инструментом `save_observation`:

```json
{
  "date": "2026-07-19",
  "observations": [
    {
      "indicator_id": 26,
      "name": "Коэффициент технической доступности, %",
      "block": "base",
      "value": 99.42,
      "unit": "%",
      "source": "письмо 07:00-07:30",
      "collected_at": "2026-07-19T07:31:12"
    }
  ]
}
```

Повторное сохранение по тому же `indicator_id` **перезаписывает** значение
(актуализация в течение дня).

### 7.3. Веб-сигналы — `data/signals_<дата>.json`

Создаётся коллектором при `fetch_web_sources`:

```json
{
  "date": "2026-07-20",
  "signals": [
    {
      "source_id": "cbr",
      "source_name": "Банк России",
      "title": "Активы на ИИС достигли 969 млрд рублей",
      "link": "https://www.cbr.ru/press/event/?id=28557",
      "published": "Mon, 20 Jul 2026 09:00:00 +0300",
      "summary": "...",
      "topics": ["finance"],
      "urgent": [],
      "collected_at": "2026-07-20T12:05:41"
    }
  ]
}
```

- `topics` — темы повестки, к которым отнесена публикация
  (`finance`, `cyber`, `security_svo`, `tech_ai`, `emergency`);
- `urgent` — сработавшие стоп-слова (теракт, кибератака, DDoS и т.п.);
  такие публикации попадают в блок «Экстренные сигналы» черновика.

Дедупликация по `link`: повторный сбор добавляет только новые публикации.

### 7.4. Конфиг веб-источников — `data/web_sources.json`

Здесь настраивается всё веб-поведение:

- `sources` — список RSS-лент. Добавить источник = добавить объект:
  ```json
  { "id": "kommersant", "name": "Коммерсантъ",
    "url": "https://www.kommersant.ru/RSS/news.xml", "type": "rss", "enabled": true }
  ```
  `"enabled": false` — временно отключить без удаления.
- `topics` — тематические словари: тема → список подстрок (ищутся в
  нижнем регистре в заголовке + тексте). Новая тема добавляется своим
  списком ключевых слов.
- `urgent_keywords` — стоп-слова для экстренных сигналов.
- `max_items_per_source` — лимит публикаций с источника (по умолч. 100).
- `request_timeout_sec` — таймаут одного источника (по умолч. 15).

### 7.5. Черновики — `data/report_draft_<дата>.md`

Готовый markdown-черновик: 5 разделов (базовый блок → динамический блок →
сигналы → веб-мониторинг → полнота данных). Всегда помечен «ЧЕРНОВИК».

## 8. Все 9 инструментов — параметры и примеры

Обращение к инструменту напрямую (для тестов без Ouroboros):

```bash
curl -X POST http://localhost:9999/mcp \
  -H 'Content-Type: application/json' \
  -d '{"jsonrpc":"2.0","id":1,"method":"tools/call","params":{"name":"<ИМЯ>","arguments":{<АРГУМЕНТЫ>}}}'
```

В чате Ouroboros достаточно сказать агенту по-человечески — он сам
выберет инструмент (примеры фраз даны ниже).

### 8.1. `list_indicators` — каталог показателей

| Параметр | Тип | Обяз. | Описание |
|----------|-----|-------|----------|
| `block` | string | нет | `base` / `dynamic` / `all` (по умолч. `all`) |

Пример: `{"name":"list_indicators","arguments":{"block":"base"}}` →
49 обязательных показателей с критериями отбора.
Фраза агенту: «Покажи показатели базового блока».

### 8.2. `get_selection_criteria` — окна отсечки и источники

Без параметров. Возвращает 53 показателя с заполненными критериями
(за какие сутки брать, когда приходит письмо).
Фраза агенту: «Когда приходят данные по IT-инцидентам?»

### 8.3. `save_observation` — загрузить значение показателя

| Параметр | Тип | Обяз. | Описание |
|----------|-----|-------|----------|
| `indicator_id` | int | да | id из каталога (1–68) |
| `date` | string | да | `ГГГГ-ММ-ДД` |
| `value` | number/string | да | значение |
| `unit` | string | нет | единица («млрд руб.», «%», «шт.») |
| `source` | string | нет | источник («письмо 06:30») |

Пример: `{"indicator_id":26,"date":"2026-07-20","value":99.87,"unit":"%","source":"письмо 07:15"}`
Фраза агенту: «Запиши: КТД за сегодня 99.87%, источник — письмо 07:15».

### 8.4. `get_observations` — что собрано за дату

| Параметр | Тип | Обяз. |
|----------|-----|-------|
| `date` | string | да |

Фраза агенту: «Что уже собрано за сегодня?»

### 8.5. `fetch_web_sources` — ЗАПУСК ВЕБ-ОБХОДА (главное)

| Параметр | Тип | Обяз. | Описание |
|----------|-----|-------|----------|
| `date` | string | нет | по умолчанию — сегодня |

Обходит все `enabled`-источники из `web_sources.json`, сохраняет новые
публикации в `data/signals_<дата>.json`. Ответ — статистика:

```json
{"new_signals": 275, "total_signals": 275,
 "sources": [{"id":"rbc","name":"РБК","fetched":30,"new":30}, ...],
 "errors": []}
```

`errors` не пустой → какой-то источник недоступен (см. раздел 11).
Фраза агенту: «Собери сигналы из открытых источников».

### 8.6. `get_signals` — выдача веб-сигналов

| Параметр | Тип | Обяз. | Описание |
|----------|-----|-------|----------|
| `date` | string | нет | по умолчанию — сегодня |
| `topic` | string | нет | `finance` / `cyber` / `security_svo` / `tech_ai` / `emergency` |
| `limit` | int | нет | по умолч. 50 |

Фраза агенту: «Покажи сигналы по кибербезопасности за сегодня».

### 8.7. `check_completeness` — контроль полноты базового блока

| Параметр | Тип | Обяз. |
|----------|-----|-------|
| `date` | string | да |

Возвращает, какие из 49 обязательных показателей ещё не поступили:
`{"complete": false, "missing_base": [{"id": 8, "name": "Остаток наличности..."}]}`
Фраза агенту: «Все ли базовые показатели собраны?»

### 8.8. `detect_hot_topics` — «горячие» темы по порогам

| Параметр | Тип | Обяз. |
|----------|-----|-------|
| `date` | string | да |

Проверяет собранные значения по пороговым правилам (`HOT_RULES` в
`mcp/server.py`): КТД < 99.5%, любые значимые инциденты, критичные
IT-инциденты, DDoS, план «Ковёр» и др.
Фраза агенту: «Есть ли горячие темы за сегодня?»

### 8.9. `build_report_draft` — СБОРКА ЧЕРНОВИКА (главное)

| Параметр | Тип | Обяз. |
|----------|-----|-------|
| `date` | string | да |

Собирает полный черновик (markdown) из 5 разделов. Ответ содержит поле
`draft_markdown` — готовый текст.
Фраза агенту: «Собери черновик справки за 2026-07-20».

## 9. Ежедневный рабочий цикл (runbook)

Тайминг привязан к расписанию источников (`memory/knowledge/sources.md`).

```
06:10  приходит письмо по кибербезопасности
06:30  письмо по наличности/УС/ВСП
07:05  IT-инциденты (окно 07:00→07:00)
07:30  КТД
08:00  инциденты безопасности
09:00  сводка СВО (окно 09:00→09:00)
09:05  ДЕДЛАЙН: базовый блок должен быть полным (Координационный Штаб)
```

### Сценарий с агентом (чат Ouroboros)

1. **Запустите сервер** (раздел 5) — один раз утром.
2. «Собери сигналы из открытых источников» → `fetch_web_sources`.
3. По мере прихода писем: «Запиши остаток наличности 512.3 млрд руб.,
   источник письмо 06:30» → `save_observation` (и так по каждому письму;
   в пилоте этот шаг парсится из почты автоматически).
4. «Проверь полноту» → `check_completeness`. Если есть пропуски —
   запросить источник или принять решение о пометке «нет данных».
5. «Есть горячие темы?» → `detect_hot_topics`.
6. «Собери черновик справки за сегодня» → `build_report_draft`.
7. **Аналитик**: читает черновик, проверяет цифры, добавляет выводы и
   интерпретацию, снимает пометку «ЧЕРНОВИК», утверждает и отправляет.
   Без этого шага материал никуда не уходит (BIBLE.md, P1).

### Повторные сборы в течение дня

`fetch_web_sources` можно запускать сколько угодно раз — добавляются
только новые публикации. Перед вечерней сводкой обновите значения через
`save_observation` и пересоберите черновик.

## 11. Типичные проблемы и решения

| Симптом | Причина | Решение |
|---------|---------|---------|
| `Address already in use` при запуске | порт 9999 занят | `pkill -f "mcp/server.py"` или поменять `PORT` в `mcp/server.py` и в конфиге Ouroboros |
| Ouroboros не видит инструменты | сервер не запущен / неверный URL | проверить `curl`-тест из раздел 5; URL должен быть `http://localhost:9999/mcp` |
| `errors` в ответе `fetch_web_sources` | источник недоступен (таймаут/блокировка) | не страшно — остальные соберутся; при постоянной недоступности поставьте `"enabled": false` в `web_sources.json` |
| `new_signals: 0` | всё уже собрано ранее | нормально: дедупликация по ссылкам |
| «— нет данных —» в черновике | показатель не загружен | загрузить через `save_observation` или оставить пометку (честнее, чем выдуманное значение — BIBLE.md P2) |
| `date должен быть в формате ГГГГ-ММ-ДД` | неверный формат даты | только `2026-07-20`, не `20.07.2026` |
| `показатель с id=N не найден` | нет такого id в каталоге | допустимые id — из `list_indicators` |
| Skill review quorum failure в Ouroboros | одна ревью-модель | добавить ключ второго провайдера (гайд, раздел 11) |
| Медленный сбор | один источник «висит» | таймаут 15 сек на источник; худший случай = +15 сек, остальные не блокируются |

## 13. Безопасность (коротко, полное — `safety/BIBLE.md`)

1. Агент **не отправляет** материалы — только человек (P1).
2. Агент **не выдумывает** значения: нет данных = пометка (P2).
3. Каждое изменение кода/правил — через ревью, откат через git (P3).
4. Неприкосновенны: `BIBLE.md`, каталог показателей, шаблон,
   пороговые правила, любые ключи/токены (P4).
5. Каждое автономное решение логируется в `memory/scratchpad.md` (P5).
6. Сеть — только утверждённые источники из `web_sources.json`;
   новый источник = подтверждение человеком (P6).
7. Режим `/evolve` запрещён; `/panic` доступен аналитику всегда (P7).
