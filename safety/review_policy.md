# Политика ревью (Safety layer, п. 9.5 гайда Ouroboros)

Настройка режима ревью для проекта: Settings → Review → **Advisory**
(рекомендация гайда для хакатона: быстрые итерации; Blocking — для
промышленного контура).

## Матрица контролов

| Контроль | Когда срабатывает | Режим в проекте |
|----------|-------------------|-----------------|
| Triad review | каждый коммит | Advisory, кворум ≥ 2 модели |
| Advisory pre-review | перед коммитом | Advisory: замечания фиксируются, не блокируют |
| Scope review | коммит, затрагивающий >1 модуля | Advisory |
| Skill review | перед выполнением навыка | Advisory + сверка хеша SKILL.md |
| Hardcoded sandbox | любая попытка записи в файлы P4 | **Blocking всегда** (не отключается) |

## Что блокируется всегда (независимо от Advisory)

- запись в файлы hardcoded sandbox (BIBLE.md, P4);
- отправка/публикация материалов без подтверждения аналитика;
- сетевые обращения вне перечня источников;
- запуск `/evolve`.

## Практика на хакатоне

1. Перед первым запуском убедиться, что настроены ≥ 2 провайдера ключей,
   иначе skill review упадёт с `quorum failure`.
2. Замечания Advisory-ревью разбираются командой в конце дня; критичные —
   немедленно.
3. Каждая стабильная итерация фиксируется: git commit → triad review →
   `promote_to_stable`. Откат — `rollback` к последнему stable.
4. После хакатона, перед пилотом в контуре Банка: перевести Review в
   Blocking и расширить hardcoded sandbox на продуктивные источники.
